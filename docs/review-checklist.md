# コンポーネントレビューチェックリスト

mitsubachi-ui（Web Components / Lit）のコードレビュー観点です。
PRレビューやセルフチェックの際に参照してください。

## レビュー観点

### 1. API設計・開発者体験
- **直感的なAPI**: プロパティ・属性・イベントの命名は分かりやすいか？必要最小限か？ — APIが多すぎると学習コストが上がり、少なすぎるとユースケースを満たせない
- **デフォルト値**: 適切なデフォルト値が設定されているか？ — 設定なしで最も一般的なユースケースが動くべき
- **カスタマイズ性**: CSSカスタムプロパティやスロットで拡張できるか？ — Shadow DOMによる完全カプセル化は、拡張手段がないと利用者を困らせる
- **ドキュメント**: TypeScriptの型定義やJSDocコメントで使い方が明確か？ — IDEの補完やStorybookのドキュメント自動生成に直結する

### 2. コンポーネント設計
- **単一責任の原則**: コンポーネントの責務は明確か？ — 複数の責務を持つと変更理由が増え、保守が困難になる
- **再利用性**: 異なるコンテキストでも使えるか？ — 特定の画面に依存した設計は再利用を妨げる
- **composability**: スロットや `part` 属性で他のコンポーネントと組み合わせやすいか？ — デザインシステムのコンポーネントは組み合わせて使われることが前提
- **制約の適切さ**: 過度に制約的でも、過度に自由でもないか？ — 制約が強すぎるとユースケースに合わず、弱すぎるとデザインの一貫性が崩れる
- **親子間のイベントハンドラ重複**: ラッパーコンポーネント（例: `*-unit`）と子コンポーネントで同じイベントハンドラが重複していないか？ — `keydown` 等の composed イベントは Shadow DOM を越えてバブルアップするため、親子両方にハンドラがあると二重実行のリスクがある。子に委譲するか、渡さない理由をコメントで明記する

### 3. アクセシビリティ
- **セマンティックHTML**: Shadow DOM内でも適切なHTML要素を使用しているか？ — `<button>`, `<dialog>` 等のネイティブ要素はキーボード操作やスクリーンリーダー対応を標準で備えている
- **キーボード操作**: キーボードだけで操作できるか？ — マウスが使えないユーザーにとって唯一の操作手段
- **ARIA属性**: 必要なARIA属性が適切に設定されているか？ — セマンティックHTMLだけでは伝えられない状態や関係性を補完する
- **フォーカス管理**: `delegatesFocus` の使用など、Shadow DOM越えのフォーカス管理が適切か？ — Shadow DOM境界でフォーカスが迷子になると操作不能になる
- **loading状態**: loading中に `aria-busy="true"` を付与しているか？ — スクリーンリーダーに処理中であることを伝える

### 4. パフォーマンス
- **不要な再レンダリング**: `@property` / `@state` の使い分けが適切か？ — `@property` は属性の変更でも再レンダリングが走るため、内部状態には `@state` を使う
- **バンドルサイズ**: 不要な依存関係を含んでいないか？ — デザインシステムは多くのプロダクトに読み込まれるため、サイズの影響が大きい
- **レンダリングコスト**: `render()` 内で重い処理を実行していないか？ — `render()` は頻繁に呼ばれるため、計算はライフサイクルメソッド側で行うべき
- **`nothing` の活用**: 条件分岐で不要なDOMを生成していないか（`nothing` を使っているか）？ — 空文字列や `undefined` と異なり、`nothing` はDOMノードを一切生成しない

### 5. Web Standards / Lit のベストプラクティス
- **Shadow DOMのカプセル化**: 外部CSSが意図せず侵入・漏洩していないか？ — カプセル化の破壊はデザインの一貫性を損なう
- **スロットの設計**: デフォルトスロット・名前付きスロットの使い分けが適切か？ — スロットはコンポーネントの拡張ポイントであり、設計がcomposabilityを左右する
- **カスタムイベント**: `CustomEvent` の `bubbles` / `composed` 設定が適切か？ — `composed: true` がないとShadow DOM外にイベントが伝播しない
- **`attributeChangedCallback` の考慮**: リフレクション（`reflect: true`）が必要なプロパティに設定されているか？ — CSSの `:host([attr])` セレクタやDOM APIでの属性取得に必要
- **`connectedCallback` / `disconnectedCallback`**: イベントリスナーの登録・解除が適切か？ — 解除漏れはメモリリークや意図しない動作の原因になる
- **スタイルの定義方法**: `*.styles.ts` に Lit の `css` タグドテンプレートリテラルで記述しているか？ — `css` タグはLitの共有スタイル機構と型チェックの恩恵を受けられる
- **HTML属性名の一貫性**: camelCaseプロパティに `reflect: true` を指定する場合、`attribute` オプションでケバブケースの属性名を明示しているか？ — 指定しないと全小文字（例: `submitbyenter`）になり可読性が下がる
- **型の厳密性**: `any` や不適切な `as` 使用を避けているか？ — 型安全性がコンパイル時のバグ検出を支える

### 6. フォーム連携
- **`formAssociated` の必要性**: フォーム内で値を送信するコンポーネントに `static formAssociated = true` と `ElementInternals` が設定されているか？ — これがないとカスタム要素は `<form>` の `FormData` に参加できない
- **`setFormValue`**: `checked` や `value` の変更時に `internals.setFormValue()` で値を同期しているか？ — 同期しないとフォーム送信時に値が含まれない
- **`requestSubmit`**: `type="submit"` ボタンがフォーム送信を正しくトリガーしているか？ — Shadow DOM内の `<button>` はネイティブのフォーム送信が効かないため、`ElementInternals.form.requestSubmit()` で明示的に送信する必要がある

### 7. CSS設計
- **CSSカスタムプロパティ**: デザイントークン（CSS変数）以外のハードコードされた値がないか？ — トークン経由にしないとデザイン変更時に追従できない
- **論理プロパティ**: 物理プロパティではなく論理プロパティを使用しているか？ — RTL（右から左）言語への対応に必要
- **擬似クラス**: `hover`・`focus` 等は擬似クラスで実装されているか？ — JS制御のクラス切り替えはブラウザの最適化が効かず、状態管理も複雑になる
- **`:focus-visible` の使用**: `:focus` ではなく `:focus-visible` を使っているか？ — マウスクリック時に不要なフォーカスリングが表示されるのを防ぐ
- **`:host` セレクタ**: コンポーネントのルートスタイルは `:host` で定義しているか？ — カスタム要素自体のスタイルは `:host` でしか制御できない
- **class名セレクタ**: IDセレクタやタグ名セレクタを使っていないか？ — Shadow DOM内でもIDの一意性は保証されず、タグ名セレクタは詳細度が低く意図しない範囲に影響する
- **レスポンシブ対応**: PCファーストで720px 1箇所のみのブレイクポイントか？ — ブレイクポイントの乱立はデザインの一貫性と保守性を損なう

### 8. 後方互換性
- **タグ名・イベント名**: `mi-*` に統一されているか？ — 命名の一貫性がプロジェクト全体の可読性を支える
- **既存エイリアスの維持**: 既存の `sp-*` / `Sp*` エイリアスを削除していないか？ — 旧タグ名を使っている利用者を壊さないため
- **public APIの維持**: 既存のプロパティ・属性・イベントを破壊的に変更していないか？ — デザインシステムは多くのプロダクトが依存しているため、破壊的変更の影響が大きい

### 9. テスト容易性
- **テスタビリティ**: ユニットテストが書きやすい構造か？ — テストしにくい構造は品質低下に直結する
- **data属性**: 必要に応じて `data-testid` 等が設定されているか？ — Shadow DOM内の要素は外部からのセレクタが限られるため、テスト用の安定したフックが必要
- **ロジックの分離**: レンダリングと関係のないロジックがコンポーネント外に切り出されているか？ — 分離されたロジックはDOMなしで高速にテストできる
- **主機能のテスト網羅**: ブランチやPRの主機能にテストがあるか？ — 特にイベントハンドラ系（キーボード操作、フォーム送信等）は、正常系・デフォルト動作・エッジケース（IME入力中など）の3観点で確認する

## レビュー手順

1. 変更されたファイルを特定する（git diff や git status を使用）
2. 各ファイルの内容を読み込む
3. **プロジェクトの既存慣習を確認する**（命名規則、ファイル構成、テストスタイル等。指摘がプロジェクトの運用方針に反していないか確認する。例: `dist/` はソース変更と合わせてコミットする運用）
4. 上記の観点でレビューする
5. 具体的な改善提案をコード例付きで示す
6. 良い点も積極的に指摘する
