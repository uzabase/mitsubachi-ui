# イベント設計方針

## 基本原則

DOM Living Standard のイベント設計ガイドに従い、以下の4つを軸とする。

1. **ネイティブ互換優先**: ネイティブHTML要素と同じように動くことを目指す
2. **意味単位で設計**: ユーザーにとって意味のある変化だけをイベントにする
3. **標準HTMLに合わせる**: カスタムイベントは原則作らない。ネイティブイベントをそのまま使う
4. **外側に必要なものだけ**: 内部で完結する処理はイベントにしない

## ネイティブイベントの扱い

- Shadow DOM 内で発火したイベントはコンポーネントのルート要素で受け取り、ルート要素自身から発火し直す
- `bubbles` / `composed` は基本 `false`。必要な場合はチームで議論して判断する
- `click` / `focus` などブラウザが `composed: true` で定義しているイベントは自動で Shadow DOM の外に届くため、再発火不要
- 二重発火が起きる場合 → `stopPropagation()` で止めてコンポーネント側で再発火

## カスタムイベントについて

カスタムイベントは原則定義しない。どうしても必要な場合はチームで議論して判断する。

- 対応するネイティブHTMLイベントが本当に存在しないか確認する
- イベント名はネイティブHTMLの命名規則に従う（プレフィックスはつけない）

## React 版との対応方針

WebComponent 側でカスタムイベントを定義した場合（チーム議論を経た場合のみ）、React 版にも対応する callback prop を必ず用意する。

- 命名: `on` + イベント名の PascalCase（例: `cancel` → `onCancel`, `change` → `onChange`）
- カスタムイベントの追加・変更時は WebComponent / React を必ず同時に更新する

## ドキュメント・Storybook・テスト

- イベントを定義した場合は `@fires` JSDoc を記載する
- Storybook の Events カテゴリに記載し、視覚的フィードバックのある Story を用意
- テストはカスタムイベントを定義した場合のみ「発火確認」「detail 確認」「cancelable 確認」の3層で行う
